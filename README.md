# Espafloria — База знаний

Репозиторий — **источник правды** для проекта Espafloria.

## Структура

```
.
├── README.md               # этот файл — правила работы
├── VERSIONS.md             # технический индекс версий файлов
├── CHANGELOG.md            # временной журнал изменений (по дате)
└── projects/
    └── espaflor/           # все файлы по проекту — плоско, без подпапок
        ├── overview.md
        ├── ...
```

Структура сознательно плоская внутри `projects/espaflor/`. Нет `processes/`, `decisions/`, `people/` — для одного проекта это избыточная иерархия. Если появится второй проект — заведём `projects/<new>/` рядом.

## Протокол версионирования

Каждый `.md`-файл начинается с маркера:

```
<!-- v: N | updated: YYYY-MM-DDTHH:MMZ -->
```

При любом изменении файла:
1. Инкрементируется `v` в самом файле
2. Обновляется `updated` (UTC, ISO 8601)
3. Синхронно обновляется строка в `VERSIONS.md`
4. В `CHANGELOG.md` добавляется запись о сути изменения
5. Всё это уходит одним коммитом (или серией последовательных — `push_files` подвисает в Docker-setup)

### Два разных файла — не дублирование

- **VERSIONS.md** — технический индекс: файл → текущая версия. Чат читает его первым при старте сессии, сверяет с Project-файлами
- **CHANGELOG.md** — хронологический журнал: что изменилось, когда, почему. Для человека, для истории

## Как работают чаты с этой базой

### При старте сессии
1. Прочитать `VERSIONS.md` (один дешёвый вызов `github:get_file_contents`)
2. Сверить версии с маркерами в Project-файлах, если они есть в контексте
3. При расхождении — **сразу** сказать пользователю, какие Project-файлы устарели

### При обновлении базы
1. Сформулировать изменение
2. Бампнуть `v` и `updated` в самом файле
3. Обновить строку в `VERSIONS.md` (и бампнуть его собственный `v`)
4. Добавить запись в `CHANGELOG.md`
5. Закоммитить
6. Сказать пользователю: «обновлено X до v N, не забудь перезалить в Project»

### Что НЕ делать
- Не коммитить файл без обновления `VERSIONS.md`
- Не инкрементировать `v` без реальных изменений
- Не создавать подпапки внутри `projects/espaflor/` — плоско

## Конвенции

- Markdown, `kebab-case.md` в именах
- Commit messages: `<раздел>: <что изменилось>`
  - `projects/espaflor: add pedido-flow`
  - `versioning: bump VERSIONS index`
- Даты — UTC, ISO 8601
