# Espaflor — База знаний

Репозиторий — **источник правды** для проектов, процессов и решений.

## Структура

```
.
├── README.md           # этот файл
├── VERSIONS.md         # индекс версий всех файлов
├── projects/           # документация по проектам
├── people/             # контекст по людям и командам
├── processes/          # процессы, чек-листы, SOP
└── decisions/          # лог решений (ADR)
```

## Протокол версионирования

Каждый `.md`-файл начинается с маркера:

```
<!-- v: N | updated: YYYY-MM-DDTHH:MMZ -->
```

При любом изменении файла:
1. Инкрементируется `v` в самом файле
2. Обновляется `updated` (UTC)
3. Синхронно обновляется строка в `VERSIONS.md`
4. Всё это уходит одним атомарным коммитом

## Как работают чаты с этой базой

### При старте сессии
1. Читать `VERSIONS.md` (один дешёвый вызов `github:get_file_contents`)
2. Сверить версии с маркерами в Project-файлах, если они есть в контексте
3. Если расхождение — **сразу** сказать пользователю, какие Project-файлы устарели и что качать

### При обновлении базы
1. Сформулировать изменение
2. Бампнуть `v` и `updated` в самом файле
3. Обновить строку в `VERSIONS.md`
4. Сделать коммит (предпочтительно `push_files` с обоими изменениями, если не виснет; иначе два последовательных `create_or_update_file`)
5. Сообщить пользователю: «обновлено X до v N, не забудь перезалить в Project»

### Что НЕ делать
- Не коммитить файл без обновления `VERSIONS.md` — индекс не должен отставать
- Не инкрементировать `v` без реальных изменений
- Не держать в Project-файлах архивные заметки — только обзорные и часто используемые

## Конвенции

- Markdown (`.md`), `kebab-case.md` в именах
- Осмысленные commit messages в формате `<раздел>: <что изменилось>`
  - `projects/espaflor: add overview`
  - `decisions: record CRM choice`
  - `processes: update morning routine`
- Даты в UTC, формат ISO 8601
