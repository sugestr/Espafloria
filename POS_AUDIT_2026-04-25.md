# POS-аудит Espafloria — что есть, чего нет, и нужно ли на Odoo.sh

**Дата:** 2026-04-25
**Контекст:** Owner просит финализировать POS-операции. Платформа сейчас — Odoo Online (SaaS) Custom (важно: 07_infrastructure_devops.md ошибочно говорит «Odoo.sh Custom» — фикс при следующей правке).
**Цель файла:** одна страница ответа «что готово / что недостаёт / куда идти / нужна ли миграция на Odoo.sh».

> ⚠️ **Этот файл — рабочий артефакт, не часть `master-context/` базы знаний.** Если после обсуждения захочется забрать сюда части — переедут в 08/09 + новый раздел в 99.

## TL;DR (3 строки)

1. **5 из 10** POS-сценариев уже в проде (стандартная продажа, eWallet-аванс, 4 букетных flow, базовый POS Ship Later); остальные 5 — гэп.
2. **Самое узкое:** photo capture в POS UI и tasks-in-POS — это **единственные** реальные драйверы миграции на Odoo.sh. Всё остальное закрывается штатно в Odoo 19 через Studio + Automated Actions + native Barcode/Quality/Pricelists.
3. **Рекомендация:** мигрировать на Odoo.sh **не сейчас**, а только когда photo-в-POS станет блокером бизнеса. Сейчас приоритет — добить штатно то, что закрывается без миграции (списание через Quality+Approvals, инвентаризация через Barcode+Cycle Counts, задания через Project+Maintenance, photo на букете через Studio Image field).

## Что закрывается без миграции (приоритет работы)

### Группа A — мелочи на 1-2 часа каждая

- **Photo на готовом букете (sale.order.line)** — добавить Studio Image поле к `sale.order.line`. На мобиле кнопка камеры — нативно через HTML5 input. ~5 минут.
- **POS Per-channel pricing (shop vs online)** — два pricelists: один на POS config, другой на website. Native в Odoo 19. ~30 минут.
- **Базовый refund flow** — кнопка Refund в POS native, проверить что работает. Тест-прогон.
- **Точка 4: букет + eWallet end-to-end** — тест: top-up 50€ → собрать букет на 80€ → settle → eWallet покрывает 50€ + cash 30€. Проверить JE и баланс карты. ~30 минут на тест.
- **Точка 5: Ship Later end-to-end** — тест: оплата сейчас, доставка через 2 дня → проверить delivery picking создаётся, qty_delivered обновляется при отгрузке.

### Группа B — на день каждая

- **Списание (точка 6) — упрощённый flow на Online:**
  1. Quality Control Point на operation type «Scrap» с типом «Take a Picture» (обязательное фото).
  2. Approvals app: тип «Stock Write-off», approver = главный флорист.
  3. Server Action на `stock.scrap.action_validate`: проверяет что есть quality.check pass + approval approved → иначе блокирует.
  4. Studio: добавить selection-поле «Тип списания» (завял/сломался/потерян/пересортица/забыт) — вписать в [05 §1.4](master-context/05_florists_logistics_accountant.md).

- **Инвентаризация (точка 9) на Online:**
  1. Включить Barcode app (если ещё не активен на Enterprise plan).
  2. Создать Cycle Count по location PLA/Stock с фильтром group=Rosas (через `pos.category` или product tag).
  3. Settings → Inventory → «Manual confirmation of adjustments» (Wait for Review) → сначала флорист считает, главный апрувит.
  4. Mobile UX через Odoo Mobile app или PWA.

- **Задания магазину (точка 10) на Online:**
  1. Project «Магазин Plaza» → recurring tasks: «помой холодильник 1×/неделю», «проверь ценники 1×/день».
  2. Task assignee = `hr.employee.user_id`.
  3. Studio Image поле «Фото-отчёт» на task (обязательное через domain).
  4. Server Action на close: блокировать если `attachment_ids = []`.
  5. Activities + automated reminder через 3 часа без действия.
  6. Maintenance app для холодильников (equipment) → maintenance request с photo proof.
  7. **POS-integration пока без custom**: на открытии смены флорист видит свой Activities widget в backend. POS-popup — следующий этап (требует Odoo.sh).

### Группа C — на полдня

- **Точка 8: receiving UX** — сделать Studio extension barcode receipt view: показать `paper_qty / expected_qty / review_color` прямо в Barcode UI. Native barcode receipts уже есть, надо вписать наши custom поля.

## Что требует Odoo.sh

| Драйвер | Жёсткость | Альтернатива на Online |
|---|---|---|
| **Photo capture в POS UI** (точка 7) | Жёсткий — нет workaround | Делать фото через Quality на picking, не через POS. Не покрывает «фото готового букета на кассе перед выдачей клиенту». |
| Task-in-POS-UI badge (часть точки 10) | Мягкий — можно через backend Activities | Флорист открывает Activities в backend на смене. UX хуже, но работает. |
| Bouquet exchange-as-one transaction (часть 3) | Мягкий — текущий v2 reserve-model работает | Текущая reserve-model уже элегантна. Не критично. |
| Полный 1-click scrap approval | Мягкий | Quality+Approvals+Server Action на Online. UX чуть хуже, бизнес-логика та же. |

**Вывод:** **из 10 пунктов миграции требует только 1 (photo в POS) жёстко**, остальное — UX-улучшения.

### Когда мигрировать?

**Не сейчас**, потому что:
1. Все 5 нерешённых пунктов (6-10) можно протолкнуть на Online через Studio + Server Actions + Quality + Approvals + Project + Barcode + Cycle Counts.
2. Миграция на Odoo.sh — **one-way** (см. 99 §26). После custom_addons обратно на Online нельзя.
3. На Odoo Online уже накоплена куча Studio-полей и Server Actions — миграция = export DB → import на Odoo.sh + проверка что всё перенеслось. Это сессия на день минимум.
4. Бизнес-блокером photo-в-POS станет, когда: (а) пойдёт регулярный поток онлайн-заказов с обязательной фотографией готового букета, (б) появятся споры с клиентами «не такое прислали». Сейчас оба — гипотетические.

**Триггеры миграции (когда — да):**
- Начнётся Flowwow / Glovo / собственный сайт с обязательной фотофиксацией готового букета в момент выдачи курьеру.
- Понадобится OCA `base_tier_validation` для multi-уровневых approvals (например, для крупных скидок в POS, manager-override flow).
- Захочется поставить custom модуль для multi-warehouse split одного albarán (P4.3 в [09](master-context/09_open_work.md)).

## Архитектура букетов — для базы знаний (краткая редакция)

> Этот раздел — кандидат на компактный summary в [05 §1.2.5](master-context/05_florists_logistics_accountant.md) или новый файл `13_bouquets.md`. Сейчас инфо разбросано по 99 §32/§46, 05 §1.2, 08 §B, 00 глоссарию. Предлагаю компактнее.

### Что моделирует

Букет — это **`sale.order` с `partner_id=53` («🌹 Букет на витрину»), `state=sale`, у которого SO-picking держит компоненты в `state=assigned`** (зарезервированы, но не списаны). Имя — `BP-YYYY-NNNN`, sequence per-warehouse.

**Почему не отдельная модель / не MRP BoM:**
- `sale.order` уже есть весь lifecycle: `sale → cancel`, attachments через chatter, line modifications, ценники, связь с POS через `sale_order_origin_id`.
- BoM пересчитывал бы каждый букет как уникальную BoM — overkill, не аналитично.
- Custom model потребовала бы свой UI, права, отчёты — велосипед.
- Reserve-model даёт правильную картинку склада: «этот цветок на витрине, в букете, уже не свободен» = `available_quantity` уменьшается, `reserved_quantity` растёт. Видно в Inventory.

### Четыре жизненных перехода

Все идут через **один payment method «🌹 Собрать / изменить букет» (id=6)** или **«🗑 Разобрать букет» (id=8)**, которые триггерят серверные actions 1203/1209.

| Триггер | Action | Ветка | Результат |
|---|---|---|---|
| Корзина без Settle + payment id=6 | 1203 | **Create** | new SO BP-*, confirm → assigned (reserve), reverse POS-picking |
| Settle BP-* + payment id=6 (с правками в корзине) | 1203 | **Reassemble** | old.cancel() → new SO BP-*, confirm → assigned, reverse POS-picking |
| Settle BP-* + любой обычный payment (cash/card/eWallet) | штатный Odoo + 1203 | **Sell** | Odoo сам cancels SO-picking, POS-picking списывает компоненты, qty_delivered++. Наш код — только chatter |
| Settle BP-* + payment id=8 | 1209 | **Dismantle** | old.cancel(), reverse POS-picking → стоки на склад |

**Маркер сборки** (product 7864 «🌹 Работа по сборке букета», service, default 5€ IVA 10%): если флорист добавил с 5€ — бонус ему за сборку; если забыл — 1203 авто-вставит с 0€ (бонуса нет, но маркер для аналитики есть).

### Почему Sell ничего не cancel'ит вручную

Подтверждено тестом 1.1 (23 апр 2026): Odoo 19 при Settle+обычный payment **штатно** cancels SO-picking, POS-picking создаёт свой move, `qty_delivered` обновляется через POS payment hook. **Условие:** SO-picking на момент Settle должен быть в `assigned`, не cancel. Старая v1 ломала это правило (cancel после confirm) — **в v2 убрано**, в этом весь рефактор.

### Деприкейт v1

Из v1 в архив: product 7865 `[BQ-DISMANTLE]`, automations 11/12, branch dismantle в 1203 (теперь в 1209), cancel SO-picking после confirm.

### Что осталось открытым (не критично)

- Pre-built букеты (утром собрал, вечером продал) формально работают через тот же flow — флорист просто через Settle открывает старый BP-* и продаёт. **Это и есть Pre-built.** Дополнительной механики не нужно.
- Photo на букете — Studio Image field на `sale.order.line` (точка H в матрице).
- Две цены (shop vs online) — pricelist на POS config + website (точка I).
- Списание завядшей розы из существующего букета — пока через Reassemble (qty=0 на завядшем компоненте). Связка с workflow «полное списание с фото» (точка 6) — будущая работа.
