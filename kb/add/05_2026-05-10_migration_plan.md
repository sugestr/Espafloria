# План миграции каталога Espafloria — снимок 2026-05-10

Snapshot структуры переезда: цели, принципы, карта блоков A1–A14, статус, принятые решения, парк-открытые. Точка возврата для следующей сессии.

## 1. Цель

Пересобрать активный каталог **снизу-вверх из реально потреблённого в 2026 году**: новый чистый набор карточек с осмысленной структурой template/variants, на нулевом стоковом остатке, с приходом сразу через апрельские pedido. Старый карантин очищается и архивируется.

## 4. Принятые решения (накопленные)

### 4.1. Атрибуты — пересобираем целиком из supplierinfo
Атрибутам, которые сидят в карточках после прошлых sub-agent runs (`x_studio_botanic_name` и др.), **не доверяем** — пересобираем целиком в A3 из:
- `product.supplierinfo.product_code` (Verdnatura codigo)
- `product.supplierinfo.product_name` (свободный текст бумаги, содержит вкрапления типа `COLOR Verde ALTURA 70 cm`)
- `product.supplierinfo.x_studio_supplier_identity_key` (Serviflor композитный ключ `SV|ART:...|UNITS_PER_PACK:...|...`)
- Текущий `name`, `default_code`, `botanic_name` — как baseline (не источник правды).

Парсинг: regex для structured-вкраплений + LLM-проход для свободного текста где regex не справляется.

### 4.2. POS-категории — зеркало учётного дерева + тематические
Базовое дерево POS = mirror `product.category`. Поверх — **псевдо-вкладки**:
- 🔥 «Акционные», ⏰ «Скоро испортится», 📦 «Текущая поставка / Прошлая поставка», 🆕 «Новинки» — **постоянно полезные, держим включёнными**.
- ❤️ «14 февраля», 🌷 «8 марта», 🎄 «Рождество», 🎃 «Хеллоуин» — **скрыты когда нет повода**, включаются за неделю-две до даты.

Динамические группы («скоро испортится» / «текущая поставка») — реализация решается в A8 (cron+toggle vs `available_in_pos`+smart-menu).

### 4.3. MIX-карты — 2-3 ценовых tier
MIX в Espafloria = **намеренная двухуровневая полка**: дорогой MIX (пример 37€) для крупных дорогих стволов, дешёвый MIX (пример 12€) для мелких. **Не сводить в одну MIX-карту.** На каждом MIX-кластере минимум 2 tier-карты (`mix_tier=cheap/standard/premium`). Применимо и к срезке (дорогой/дешёвый цветок).

**Особенность:** MIX часто **не подписан как MIX в имени**. Детектим по «много разных supplier-имён сходится на одну карту в pedido lines».

### 4.4. Фото — двухуровневая галерея
- **Главное `image_1920`** — supplier-фото (красивое). Это видит флорист на POS-тайле.
- **Галерея `product.image`** — наше receipt-фото («что физически приехало») + остальные supplier-фото если их несколько.

**Технический момент:** на текущей платформе (Odoo Online Custom) **прямой URL→binary fetch невозможен** (вердикт коммита `7d5d0cc`: safe_eval запрещает import requests, base_import ждёт base64, custom widget на Online недоступен). Варианты:
1. Внешний скрипт по паттерну `add/08_fetch_holded_images_55.py` (download + resize + MCP `set_binary_field`).
2. UI Import Records с external_id + поле `image_1920` как URL — **не проверено** на 19.0, нужен 1-card test в A8.
3. Make.com pre-fetch URL→base64 при создании pedido draft.

В A8 короткий тест варианта 2 → если работает, идём bulk-import; если нет — fallback на внешний скрипт.

### 4.5. Цены — `list_price` берём из текущего Holded
**Holded = источник истины** для цен продажи. Получили полный экспорт Holded products (2051 строк) с обоими представлениями (Subtotal без НДС / Total с НДС) — будет основа для A9.

5 статусов ревью (на новых картах после миграции):
- `accept_old_ok` — старая цена, рентабельность норм
- `review_too_cheap` — продаём слишком дёшево, проверить
- `review_too_expensive` — слишком дорого, проверить
- `new_x3_placeholder` — новая карта, цена ×3 от закупки заглушкой
- `verified` — проверено, дата + автор зафиксированы

Технически: `x_studio_price_review_status` (selection) + `x_studio_price_reviewed_by` (m2o) + `x_studio_price_reviewed_on` (date).

### 4.6. VAT switch (без НДС → с НДС)
Сейчас система работает в режиме «цены без НДС» — неудобно для POS (клиент видит с НДС) и логисту (сложно подгонять круглые суммы 35.00 / 34.95 при переключении контекста). Переключение **разовое через UI настройки**, требует:
- Бекап перед переключением.
- Проверка как Odoo 19 интерпретирует существующие `list_price` (пересчёт vs смена интерпретации) — **уточнить с docs.odoo.com/19.0**.
- Тест на одной карте до bulk.
- Проверка как наши маржинальные поля себя ведут (они оперировали ценами без НДС, могут так и остаться).

**Когда переключаем — открытый вопрос (парк):** до создания target skeletons (вся миграция уже на новой семантике) или после миграции перед открытием продаж. Решение в A9-pre.

### 4.7. Bill control (`purchase_method`) — решено
| Категория товара | Режим | На пальцах |
|---|---|---|
| Срезка + горшечка (~880 карт) | `purchase` (On ordered) | Платим бумагу. ±2-3 стебля списываются молча через recount |
| Декорация / упаковка / расходники / оборудование / доставка (~650 карт) | `receive` (On received) | Платим факт. Если приехало меньше — счёт автоматом меньше |

«Ругаться с поставщиком» — **отдельный механизм**, не bill_control. Это:
1. Маркировка приёмки (`review_status` краснеет при расхождении >5 стеблей, [03 §4](../03_inventory_pipeline.md)).
2. Бухгалтер при серьёзном косяке либо правит счёт руками, либо принимает + получает credit note.

Bulk-write на ~880 карт — **в блоке A7** после миграции.

### 4.8. Печать ценников — автоматическая задача магазину
Когда логист меняет цену или создаёт новую карту → автоматически рождается задача магазину «эй Gloria/Blau, у вас есть этот товар, печатать/переклеить». Магазин отмечает выполнение в системе.

**Механика — открытый вопрос (парк):** через `mail.activity` (привязка к карте, у юзера-Gloria/Blau появляется TODO) vs `project.task` (отдельный board «ценники к печати») vs `maintenance.request`. Решение в A10.

### 4.9. Скоуп миграции
- **В этом проходе:** только срезка + горшечка (трек α).
- **Твёрдый товар (трек β):** отдельным треком после α (блок A13). 1:1 миграция, без реструктуризации, остатки-константа из Holded.
- **Служебные карты (трек γ):** не трогаем.

### 4.10. Поэтапная коррекция дизайна после запуска
После открытия каталога **могут вылезать ошибки группировки** («ой, эту розу мы зря в Тюльпаны MIX кидали — надо отделить»). Нам нужен механизм коррекции уже после миграции — это **отдельный sub-track** в эксплуатации, не блок миграции.

## 9. Workflow обновления документа

При закрытии каждого блока (Aₙ) в этом документе:
1. Поменять статус блока с ⬜ на 🟢 + одна строка результата.
2. Обновить §5 «Что уже сделано».
3. Если в блоке всплыли новые решения — добавить в §4.
4. Если в блоке появились новые открытые вопросы — добавить в §6.
5. Bump `v` в header + дата.
6. Запись в `CHANGELOG.md`.
