При создании `sale.order` для букета (в т.ч. WhatsApp/online) — **компоненты пишутся по своим обычным ценам**. Подгон под итоговую сумму — через **общую скидку %** на SO или через корректирующую строку, **НЕ через price=0 на компонентах**.

**Why:** Андрей сказал «компоненты в 0€ ломают статистику маржинальности» — теряется per-item revenue, средняя цена розы становится ложной, маркетинг и закупки получают мусорные цифры для решений.

**How to apply:**
- Любой SO с компонентами букета (offline reserve-model, online WhatsApp, marketplace) — компоненты по своему `list_price` или `pricelist_price`.
- Скидка под итог — `sale.order.discount` или общая `% discount` на финальной строке-корректировке.
- Помнить [99 §G11](master-context/99_invariants.md): после `create([{...,'discount':X}])` сделать `.write({'discount': X})` — pricelist onchange перезатрёт.
- Маркер сборки 7864 = отдельная строка по своей цене (5€ если бонус флористу, 0€ только в исключении когда auto-вставлен post-factum за «забыл»).
