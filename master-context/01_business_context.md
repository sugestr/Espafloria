## Цель внедрения Odoo

Odoo должен стать **единой операционной системой цветочного бизнеса**, не «заменой Holded» и не «бухгалтерской программой».

Объединяет:
- Закупки и приёмку
- Многоскладской учёт
- POS-продажи
- Сборку букетов (производство)
- Доставку
- Online / marketplace-каналы (Flowwow, Glovo)
- Контроль ошибок персонала
- Данные для бухгалтерии Испании
- Автоматизацию через Make.com + OpenAI + Python

**Ключевая установка:** Odoo = **источник операционной правды**, а не «место, куда бухгалтер задним числом вбивает цифры».

## Три источника истины

Один товар имеет **три независимых взгляда**:

| # | Источник | Где живёт | Когда появляется |
|---|---|---|---|
| 1 | **Бумага поставщика** | `purchase.order.line.product_qty` + `price_unit` | При создании pedido (OCR от бота или вручную) |
| 2 | **Оценка логиста** | `purchase.order.line.x_studio_expected_qty` | Опционально, логист пересчитывает на приёмке/в пути |
| 3 | **Факт флориста** | `stock.move.quantity` + `x_studio_received_packs` | При физической приёмке в магазине |

**Правила:**
- Бумаге поставщика **нельзя слепо доверять**.
- Логист = подсказка, не факт.
- **Receipt / stock.move = источник складской правды.**
- Все три хранятся независимо, **одно не перезаписывает другое**.

### Bill / Invoice / Payment слой

| Сущность | Что означает |
|---|---|
| `purchase.order.line.product_qty` | Бумага (paper) |
| `qty_received` (штатное Odoo) | Сколько реально приняли (= сумма stock.move.quantity) |
| `qty_invoiced` (штатное Odoo) | Что попало в vendor bill |

При `purchase_method = 'purchase'` (цветы/горшечка) bill ждёт `product_qty` — то есть платим по бумаге. Расхождения видны в виде:
- `product_qty - qty_received` = недопоставка
- `product_qty - qty_invoiced` = «бумага vs бухгалтерия» (должно быть 0)
- `qty_received - qty_invoiced ≠ 0` = бухгалтер не всё провёл

## Pack vs Unit duality

**Ключевая специфика:** товар часто приходит в пачках, но количество единиц в пачке **нестабильно**.

- 1 pack Mimosa сегодня = 3 стебля
- 1 pack Mimosa завтра = 7 стеблей
- Розы обычно стабильны (20-25 шт/пачка), но бывает недовес

**Решение:**
- Не использовать UoM-коэффициенты как business truth
- UoM в Odoo = **только label** («Units» / «Paquete»)
- Все расчёты — через кастомные поля и Python-compute
- Связка pack ↔ units вычисляется **в момент приёмки**, не заранее

**Формула effective unit cost (для будущего):**
```
effective_unit_cost = paper_total_amount / actual_received_units
```
Пример: 4 пачки по 20 € = 80 € paper_total, флорист насчитал 52 стебля → 1.538 €/стебель.

> 🔴 **CONCEPT** — catch-weight формула и compute stock valuation не реализованы. Это будущий custom module.

## User Stories (пользовательские сценарии)

Ниже — 14 кейсов, ради которых строится система.

> ℹ️ Нумерация «4.x» — legacy (остаток от прежней структуры файла). Ссылки на эти user stories встречаются в других .md и в git-истории — не перенумеровывать.

### 4.1. Ретроспективный импорт albarán из Holded
**Что:** бухгалтер вносит старый документ вручную, доводит до состояния «правда».
**Flow:**
1. Создать / импортировать `purchase.order`
2. Привести строки к бумаге поставщика
3. Опционально заполнить логистический пересчёт
4. Создать receipt из pedido
5. Зафиксировать приёмку
6. Позже по месячной factura создать vendor bill

### 4.2. Логист закупил, флорист принимает
**Что:** приёмка независима от логиста и бумаги.
**Flow:**
1. Логист создаёт pedido с бумагой + единицей + (опц.) expected units
2. Флорист открывает receipt
3. Вводит реальные пачки и реальные штуки
4. Система показывает расхождения

### 4.3. Логист заранее видит, что бумага врёт
**Пример:** на бумаге 20, логист видит 25.
**Что делает:** заполняет `x_studio_expected_qty = 25`, бумагу не трогает. Флорист видит подсказку.

### 4.4. Пачки с неизвестным содержимым
**Пример:** бумага 10 packs, логист ожидает 49 штук, флорист насчитал 118 штук.
**Что показать в receipt:** все 4 числа — paper qty, paper unit, expected units, actual packs + actual units.

### 4.5. Один albarán — несколько магазинов
**Пример:** 50 роз → 20 Plaza + 30 Gloria.
**Требование:** сохранить связь с одним документом поставщика, разнести по складам.
> 🔴 **CONCEPT** — multi-warehouse split не реализован.

### 4.6. Букет собран заранее, выставлен на витрину
**Flow:**
1. Флорист собирает букет → компоненты списываются
2. Появляется «готовый букет» как отдельный продаваемый объект
3. Задаётся цена → печатается ценник
4. Позже: продажа / скидка / доставка / изменение состава
> 🔴 **CONCEPT** — production/BoM модуль не реализован.

### 4.7. Букет собирается под клиента на месте
**Flow:**
1. Флорист идёт в холодильник, выбирает/сканирует цветы
2. Собирает букет, добавляет труд
3. Может округлить цену
4. Добавляет доставку, закрывает чек
**Требование:** не заставлять делать line discount по каждой строке.
> 🟢 **IMPLEMENTED 2026-04-23** — Made-on-sale сборка, модификация (reassemble) и разборка через POS payment method «Собрать букет» (id=6). SO `BP-YYYY-NNNN`, tech partner Anon id=53, маркер разборки product id=7865. См. [05 §1.2.2–1.2.3](05_florists_logistics_accountant.md) и [99 §46](99_invariants.md).

### 4.8. Витрина vs online — разные цены
**Пример:** витрина 37 €, online 45 €.
**Требование:** не создавать отдельные товары, использовать pricelist rules.
> 🔴 **CONCEPT** — pricelist multi-channel не настроен.

### 4.9. Доставка — отдельная сущность
**Правила:**
- Скидка применяется к букету, **не** к доставке.
- Доставка — отдельная `sale.order.line` с отдельным product.

### 4.10. Flowwow / посредники
**Модель:**
- Клиент платит площадке
- Площадка списывает комиссию
- Переводит остаток

**Учётная трактовка (решено 2026-04-19, см. [99 §35](99_invariants.md)):** все marketplace — **юридически посредники**, клиент — наш. Комиссия платформы = наш расход, НЕ скидка клиенту. Клиент, комиссия, перевод — три независимые проводки.

### 4.11. Ошибки флористов при продаже
**Примеры:** белую розу пробили как красную; забыли списать порчу; пересортица.
**Требование:** фиксировать ошибку, **не** переписывать историю задним числом, давать аналитику.
> 🔴 **CONCEPT** — error tracking не настроен.

### 4.12. Фото готового букета к продаже
**Цели:** контроль качества, разбор конфликтов, аналитика работы флористов.
> 🔴 **CONCEPT** — photo attachments к sale.order не настроены.

### 4.13. Планшеты + личные телефоны
**Требование:** стационарный планшет на точке + возможность работы с личного телефона без полноценных Odoo-лицензий.
**Решение (будущее):** POS общие с привязкой продажи к **смене сотрудника**, не к Odoo-user.

### 4.14. Перепечатка ценников
**Требование:** система подсвечивает изменение цены, ценник печатается на термопринтере, брендированный.
> 🔴 **CONCEPT** — label printing pipeline не настроен.

## Управленческие принципы

1. **Receipt = главный операционный слой истины.** Не PO, не Invoice.
2. **Бумаге поставщика нельзя слепо доверять.** Хранится, но не истина.
3. **Логистическая информация = подсказка, не факт.** `expected_qty` не перезаписывает факт приёмки.
4. **Пустое важнее ложного нуля.** Если логист не считал — не показывать `0`, показывать пусто (отсюда sentinel `-1`).
5. **UoM = технический слой, не бизнес-истина.** Особенно для пачек.

## Бизнес-стратегия запуска (принята 2026-04-18)

**MVP 20 апреля 2026 — параллельный режим:**
- Odoo и Holded работают параллельно
- Постепенный переход операций в Odoo
- Исторические данные затягиваем за 2026 год с 1 января
- Финальный cutover — 1 января 2027

**Порядок физической миграции:**
1. Всосать все albaran из Holded (180+ документов)
2. Догрузить вручную недостающие
3. Всосать все продажи
4. Сформировать новый структурный каталог
5. Мигрировать карточки из карантина в новые variants
6. Разрешить продажу из нового каталога
7. Настроить рабочие места для ролей
8. Запуск магазинов

**Источник правды по остаткам при запуске:**

| Тип товара | Источник |
|---|---|
| Срезка (FLORES CORTADAS) | **Физический пересчёт** |
| Горшечная (PLANTAS EN MACETAS) | **Физический пересчёт** |
| Вазы, декор, упаковка, сервисы | Остатки из Holded + rolling correction: при **первой продаже** вазы — обязательный пересчёт именно этих ваз в этом магазине |

## Классы ошибок для аналитики

Система должна различать:
- Ошибка поставщика (недопоставка, пересортица)
- Ошибка логиста (неверный пересчёт в пути)
- Ошибка флориста при приёмке
- Ошибка флориста при продаже / списании
- Нерегистрированная порча / «усушка»
- Расхождение в инвентаризации
