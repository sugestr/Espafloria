# 10. Vision & Strategic Roadmap

**Последнее обновление:** 2026-04-18 (со слов Andriy)

---

## 🎯 Конечная цель

> **Умная сеть цветочных магазинов**, где база данных и роботы (автоматизация) берут на себя основную часть работы по:
> - Контролю работы сотрудников
> - Структурной целостности бизнеса
> - Выявлению слабых мест — от цен до халатности
>
> Это позволяет **экономить на управляющем персонале**, который не работает с клиентом, а только контролирует. Сэкономленные деньги идут на **мотивацию полевых сотрудников** (флористы, продавцы, курьеры).

**Ключевая метрика успеха (неявная):** соотношение людей-в-поле к людям-в-офисе должно расти в пользу поля.

## 🧭 Стратегический roadmap — 15 шагов

**Важно:** этот документ — **видение**, не оперативный TODO. Оперативная работа и приоритеты — в [09_open_work.md](09_open_work.md).

### Шаг 1. Минимальный UI для флориста (и возможно логиста)

**Операционный цикл: заказ → привезли → посчитали.**

Флорист должен удобно делать на планшете:

**Базовые операции:**
- Приёмка товара с review-статусами
- Ввод пачек и штук
- «Посчитать в магазине» кнопка
- Проверка расхождений с бумагой и логистом

**Следующий уровень — «умные задания»:**
- «Тебе пришло задание пересчитать розы»
- «Задание пришло 3 часа назад, ты ещё не пересчитал → напоминание»
- Задачи уборки / контроля: «помыть холодильник + фото-отчёт», «пересчитать все розы», «проверить ценники на товарах»
- Автоматическое отслеживание выполнения задач

**Геймификация:**
- «Хорошо продаёшь — получи премию»
- Автосчёт часов работы
- Личные показатели (performance dashboard)

**Текущий статус:** 🔴 CONCEPT. Есть базовая приёмка (см. [03_odoo_receipt_review.md](03_odoo_receipt_review.md)), но «умные задания» — это отдельный custom module или app.

### Шаг 2. Отдельный учёт флористов без лицензий

Каждый сотрудник должен считаться **независимо**, но без создания полноценных Odoo-лицензий.

**Решение:**
- Авторизация через планшет по **PIN-коду**
- Привязка продажи/приёмки/задачи к сотруднику через смену (`hr.attendance` или custom)
- Штатный `hr.employee` без `res.users` записи
- POS-кассир = employee, не user

**Статус:** 🟡 READY — у Odoo есть `POS Employees` feature, надо включить и настроить.

### Шаг 3. 🚀 ТОЧКА ЗАПУСКА В ПРОД

**Старт MVP = когда выполнены:**
1. **Инвентаризация сделана** — по цветам и горшечке физический пересчёт, по вазам/декору остатки из Holded
2. **Каталог выбран:**
   - **План A (предпочтительный):** новый каталог из товаров 2026 года, продаём на нём
   - **План B (резервный):** если новый каталог не успеваем → разблокируем карантин массовым `sale_ok=True`
3. **Флорист может принять товар** (receipt UI готов)
4. **Продавец может продать** при реальных остатках на складе

**Чего НЕ делаем в MVP (вопреки первоначальной идее):**
- ❌ **Не разрешаем «продажи в минус» как норму.** Продажа в минус — **индикатор проблемы процесса**, не режим работы. Если товара нет — значит не принят / не учтён / потерян → разбор, не продажа с дефицитом. См. [инвариант 37](99_invariants.md).
- ❌ Не запускаем без минимальной инвентаризации — это гарантированно создаст хаос

**Принцип:** лучше **задержать** запуск на пару дней и сделать нормальную инвентаризацию, чем запустить «с минусами и разберёмся потом». Разбор потом займёт больше времени, чем инвентаризация сейчас.

**Статус:** 🎯 **цель на 20 апреля 2026** (если не сдвиг).

### Шаг 4. Инвентаризация

Два варианта:
- **Поточная** (rolling): считаем по мере продаж
- **Сплошная** (full physical): разом все 3 магазина

**Решение (принято 18 апреля):**

| Товар | Метод |
|---|---|
| Срезка (FLORES CORTADAS) | Сплошная физическая |
| Горшечная (PLANTAS EN MACETAS) | Сплошная физическая |
| Вазы / декор / упаковка / сервисы | Остатки из Holded + rolling correction при первой продаже |

**Статус:** ⬜ TODO до запуска магазинов.

### Шаг 5. Догнать закупки + перестроить каталог

**Параллельно с работой:**
1. Массовый импорт albaran за 2026 (180+ документов) — в понедельник 21 апреля сотрудник
2. Перестройка нового каталога → **на вариантах**, опираясь на то, что реально закупали в этом году
3. Мигрировать карточки из карантина в новый каталог по мере закупок

**Принцип:** новый каталог строится от **реальной закупочной практики**, а не от идеального справочника.

**Статус:** 🟡 READY (инфраструктура готова), 🔴 not started (работа впереди).

### Шаг 6. Догнать продажи

- Экспорт продаж из Holded за 2026
- **Матчить на старый каталог** (карантинные или архивные карточки)
- Даже если карточка уже мигрировала → всё равно продажа садится на **старую**

**Цель:** сохранить целостность исторических данных, не смешивать новый каталог со старым.

**Статус:** 🔴 CONCEPT, start позже (май?).

### Шаг 7. Догнать фактуры + свести contabilidad

- Все factura за 2026, включая Verdnatura-специфика (товары отдельно в albaran, цены в factura)
- Свести итоги бухгалтерии → сверить с Holded totals
- Цель: учётная правда за весь 2026 год в Odoo должна совпадать с Holded

**Статус:** 🔴 CONCEPT.

### Шаг 8. Банки + касса + сверка

- Подключить банковские счета через `account.online.link`
- Свести всю банк+наличку **с 1 января 2026**
- Сверить итоги с Holded

**Статус:** 🔴 CONCEPT.

### Шаг 9. Автоматизация рабочего места бухгалтера

Три задачи бухгалтера (детали в [05_florists_logistics_accountant.md](05_florists_logistics_accountant.md)):

1. **Сверка продаж** (система × Telegram × касса × банк)
2. **Расходные накладные + испанская категоризация** (Modelo 303/347/349)
3. **Супервайзинг цикла** «закупили → привезли → распаковали → ценники → витрина → продажа»

**Цель:** убрать ручную работу в Google Sheets, всё видеть в Odoo UI.

**Статус:** 🔴 CONCEPT, 🟡 READY (локализация `l10n_es.*` уже установлена).

### Шаг 10. Holded → архив

После 9-го шага:
- Holded остаётся **только как архивная система**
- Вся оперативная работа — в Odoo
- Holded держим для исторических отчётов + compliance до аудита

**Целевая дата:** 1 января 2027.

### Шаг 11. Аналитика и оптимизация

**⚠️ Предпосылка:** точная аналитика возможна **только после миграции каталога** (шаги 5-6). До этого данные смешаны между старыми карантинными и новыми variants → отчёты «год к году» будут врать первый год.

**Для закупщика:**
- Какие товары дают наибольший margin
- Поставщики: кто недопоставляет, завышает цены
- Оборачиваемость (особенно цветы — скоропорт)
- Потери: вялый / списан / пересортица / нереализован

**Для маркетолога:**
- Популярные товары по точкам
- Сезонность
- Воронка конверсии online → offline
- Клиентская аналитика (см. **[11_crm_and_customers.md](11_crm_and_customers.md)**)

**Статус:** 🔴 CONCEPT.

### Шаг 11.5. 🆕 CRM и клиенты (параллельный workstream)

Построение CRM — **не отдельный этап, а параллельный поток**, работающий с шага 3 (MVP запуск).

**Базовые фичи с запуска:**
- Создание клиента при продаже по телефону
- История покупок привязана к `partner_id`

**Следующие этапы (по 2-3 месяцам данных):**
- Сегментация (VIP / Regular / Dormant / New)
- Email + WhatsApp рассылки
- Программа лояльности
- Напоминания о значимых датах

**Финальное:**
- Полная omnichannel-воронка (POS + Flowwow + Glovo + WhatsApp + Instagram + сайт)
- Персональные промо на базе AI-рекомендаций

Подробно: **[11_crm_and_customers.md](11_crm_and_customers.md)**

### Шаг 12. Marketplace: Flowwow / WhatsApp / Glovo

**Ключевое требование:** **премирование сотрудников от обработанной работы** → система должна отслеживать, **кто обрабатывал заказ**.

**Три канала:**
- **Flowwow** — комиссионная модель цветочной площадки
- **WhatsApp** — заказы от постоянных клиентов прямо в чат
- **Glovo** — delivery marketplace с курьерскими окнами

**Что нужно:**
- Единая воронка входящих заказов
- Привязка заказа к обрабатывающему сотруднику
- Расчёт премии пропорционально обработке
- Разделение: продажа клиенту / комиссия площадки / cash movement

**Статус:** 🔴 CONCEPT, нет брифа — требует отдельной сессии.

### Шаг 13. Веб-сайт на Odoo

Собственный сайт Espafloria на Odoo Website:
- Каталог товаров
- Онлайн-заказы
- Интеграция с POS и inventory

**Статус:** 🔴 CONCEPT.

### Шаг 14. Покупки в один клик

Интеграция:
- Instagram → Odoo (через Shop)
- WhatsApp → Odoo
- Flowwow → Odoo
- Свой сайт → Odoo

**Цель:** клиент заказывает в один клик, заказ сразу попадает в Odoo с привязкой к обрабатывающему сотруднику.

**Статус:** 🔴 CONCEPT.

### Шаг 15. 🤖 Отказ от Telegram как логирующего механизма

**Идея: Odoo становится единственной системой контроля работы.**

Вместо чатов в Telegram:

**А. Автоматический контроль (auto-detection):**
- **Фото букета** перед продажей → автоматом в базу, привязано к sale.order.line
- **Несхождения в процессе** — система сама находит и поднимает alert:
  - Наличная касса vs система
  - Не тот товар списан
  - Не та цена пробита
  - Забыли взять деньги
  - Не выполнена типовая задача к сроку
  - Не прислан фото-отчёт (холодильник, ценники и т.д.)

**Б. Системные задания (scheduled tasks):**
- «Пересчитай все розы к 14:00»
- «Помой холодильник + пришли фото»
- «Проверь наклеены ли ценники на витрину» → менеджеру приходит фото-отчёт

**В. Обнаружение халатности:**
- Автоматические сигналы при отклонениях
- Dashboards для менеджера с красными флажками
- Периодические sampling-проверки

**Статус:** 🔴 CONCEPT — это самая амбициозная часть vision.

## 🎯 Принципы реализации

Из ответов и решений по ходу проекта:

1. **«Побыстрее в бизнес»** — лучше MVP с недостатками, чем идеал в мечтах
2. **Не «беги так быстро»** — но и не тормозить без причины; баланс
3. **Делегирование быстрых операций** — если что-то проще сделать в Excel / UI вручную, чем через API, делаем вручную
4. **CHANGELOG обязательно** — каждое изменение фиксируется
5. **Один источник правды** — не плодить разные чаты разных AI; Master Context = единое место
6. **Контроль vs работа** — автоматизация контроля, чтобы люди работали, а не контролировали
7. **Мотивация в поле** — деньги должны идти тем, кто реально создаёт ценность для клиента

## 🚧 Технические предпосылки для достижения vision

Для шагов 11-15 (умная сеть) потребуется:

- **Custom Odoo modules** (не только Studio) — для сложных сценариев типа catch-weight, marketplace integrations
- **ML-компоненты** — fraud detection, demand forecasting, optimal pricing
- **Хорошая IoT-интеграция** — весы, термопринтеры, холодильники с датчиками температуры
- **Mobile-first UX** — всё через планшеты и телефоны, не десктоп
- **API-first архитектура** — чтобы любая третья сторона могла интегрироваться

Эти предпосылки — не сегодняшняя задача, но держим в уме при каждом архитектурном решении. Не заезжать в тупиковые решения.
