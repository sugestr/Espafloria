# 05. Роли и рабочие процессы

Статус: 🔴 **CONCEPT / PARTIAL** — функциональные spec-и описаны, реализация UX частично.

---

## Общая модель лицензирования ролей

**Принцип:** минимум полных Odoo-лицензий, максимум работы через планшет + PIN.

| Роль | Odoo-доступ | Устройство | Авторизация |
|---|---|---|---|
| **Флорист (продавец-сборщик)** | Без личной учётки | Планшет на стене + свой телефон + баркод-сканер (приоритет) | PIN + смена |
| **Главный флорист / Manager** | Жена + нанятый директор — **полный backend** ИЛИ extended PIN на планшете | Десктоп / планшет | Полная учётка ИЛИ PIN |
| **Логист** | Полная учётка | Планшет / телефон / десктоп | Odoo credentials |
| **Бухгалтер** | Полная учётка | Десктоп | Odoo credentials |
| **Владелец / аналитик** | Полная учётка + admin | Всё | Odoo credentials |

**Шифт-модель:**
- Сотрудник регистрируется на начало смены (`hr.attendance` или custom)
- На POS в выпадающем списке продавцов — **только те, кто на смене**
- Продажа / обработанный онлайн-заказ / приёмка / списание → привязано к смене конкретного сотрудника

## 1. Рабочее место флориста

**Главный принцип:** флорист — не «пользователь Odoo», а **оператор быстрых процессов**. Интерфейсы максимально упрощены, адаптированы под мобильные устройства, ориентированы на сканирование и фото.

### 1.1. Продажа (POS)

**Приоритет устройств:**
1. Баркод-сканер (особенно в холодильнике)
2. Личный телефон флориста
3. Планшет на стене

**Требования:**
- Минимум кликов, высокая скорость
- UI близок к POS, не к классическому Odoo backend
- Продажа через сканирование штрихкода — основной сценарий
- При ручном поиске — быстрая подсказка по первым символам (default_code / name / barcode)

**Кто продал:**
- Привязка к сотруднику **обязательна** (основа для бонусов)
- Выбор из списка **только работающих сейчас** смен

**Открытые вопросы:**
- ⬜ Единый режим для быстрых продаж vs отдельный режим для букетов vs отдельный для доставки — не финализировано
- ⬜ Стационарный планшет **основной** или равноправен с телефонами?

### 1.2. Букет как отдельная сущность

**Принципы (из [99 §32](99_invariants.md)):**
- Букет = **отдельная запись учёта**, не просто набор товаров
- Может быть собран и **не продан сразу** (витрина), потом продан
- Может быть модифицирован / разобран после сборки

#### 1.2.1. Создание букета (Create)

🟢 **IMPLEMENTED 2026-04-23 v2 (reserve-model) через action 1203 branch create.**

Флорист-флоу:
1. В POS сканирует/пробивает компоненты (розы, зелень, упаковка).
2. Опционально жмёт кнопку **«🌹 Работа по сборке букета»** (product 7864, service, 5€ default с IVA 10%). Если забыл — скрипт сам добавит в SO с price=0 (без бонуса флористу).
3. Корректирует финальную цену букета если нужно (ручная правка строк).
4. Payment → метод **«🌹 Собрать / изменить букет»** (id=6) → Validate.

Серверная логика (action 1203 branch create):
- Создаёт SO `BP-YYYY-NNNN` с `partner_id=53` (Anon), warehouse из config, все POS-линии + auto-marker если нужен.
- `action_confirm()` → SO-picking встаёт в **`assigned`** (reserve).
- **НЕ cancel SO-picking** — в этом суть reserve-model.
- Reverse POS-picking (которое Odoo создал и списало) → компоненты возвращаются на склад, где их держит reserve нового SO.
- Chatter в SO: «🌹 Собран <флористом> через POS-NNN. Состав: ...»

**Результат на складе:** компоненты **не списаны**, а **зарезервированы**. `stock.quant.quantity` не изменился, `reserved_quantity` += 1 на каждый компонент. `available_quantity` уменьшился — другой чек не заберёт.

**Цена:**
- Система предлагает цену (сумма компонентов + маркер сборки + наценка)
- Флорист может корректировать вручную
- Скидка — на уровне букета, не на строках компонентов ([99 §32](99_invariants.md))

#### 1.2.2. Модификация букета (Reassemble)

**Реальный сценарий:** *«Другой флорист взял уже готовый букет, добавил пару роз, одну розу списал как завявшую, потом вернул на витрину.»*

🟢 **IMPLEMENTED 2026-04-23 v2 через action 1203 branch reassemble.**

Флорист-флоу:
1. В POS: **Orders → Settle BP-*** → в корзину падают все линии с `sale_order_origin_id`.
2. Правит в корзине: qty=0 на убираемом компоненте, скан нового компонента, ручная правка цены.
3. Payment → метод **«🌹 Собрать / изменить букет»** (id=6) → Validate.

Серверная логика (action 1203 branch reassemble):
- Находит old SO через `sale_order_origin_id` (partner=53, state=sale).
- `old.action_cancel()` → SO штатно cancelled, его SO-picking тоже cancel (reserve отпустилась).
- Создаёт **new** SO `BP-YYYY-NNNN` с текущим составом.
- `action_confirm()` → new SO-picking assigned (новый reserve).
- Reverse POS-picking → стоки возвращаются, new reserve держит.
- Chatter: old «✏️ Пересобран в <new>», new «✏️ Пересобран из <old>».

**Net stock:** reserve перевесился со старого BP на новый. Убрал розу — она не в reserve, free. Добавил вазу — она теперь в reserve.

#### 1.2.3. Продажа букета клиенту (Sell)

**Сценарий:** клиент пришёл за готовым букетом с витрины, или флорист слегка подправил состав и сразу продал известному клиенту.

🟢 **IMPLEMENTED 2026-04-23 v2 через штатный Odoo + action 1203 branch sell (только chatter).**

Флорист-флоу:
1. В POS: **Orders → Settle BP-*** → в корзину падают компоненты.
2. Опционально правит состав (добавил розу, убрал упаковку).
3. Опционально выбирает known-клиента для CRM / LTV.
4. Payment → **Cash / Card / eWallet / etc** (любой обычный метод, **НЕ** «Собрать / изменить букет») → Validate.

Серверная логика (**штатная Odoo 19, не наш код**):
- Odoo cancels SO-picking старого BP (он был assigned).
- POS-picking этого чека списывает компоненты через свой move (реальная отгрузка).
- `sale.order.line.qty_delivered` обновляется на совпавших линиях.
- `sale.order.invoice_status=invoiced` (через POS payment hook).

Наш action 1203 branch sell **только** пишет chatter в old SO: «💰 Продан <флористом> клиенту <partner> через POS-NNN. Итого X€.»

**Net stock:** компоненты физически ушли клиенту (списаны ровно 1 раз через POS-picking, без двойного списания — потому что SO-picking был в assigned, а не cancel). Подтверждено тестом 1.1 ([CHANGELOG 2026-04-23](CHANGELOG.md)).

#### 1.2.4. Разборка букета (Dismantle)

**Сценарий:** букет завял/не продался/клиент отказался, вернуть компоненты в наличие.

🟢 **IMPLEMENTED 2026-04-23 v2 через отдельный action 1209 и отдельный payment method.**

Флорист-флоу:
1. В POS: **Orders → Settle BP-*** → компоненты в корзине.
2. Payment → метод **«🗑 Разобрать букет»** (id=8) → Validate.

Серверная логика (action 1209):
- Находит old SO через `sale_order_origin_id` (partner=53, state=sale).
- Если не нашёл — `raise UserError` («нет связи с активным букетом»).
- `old.action_cancel()` → SO и SO-picking cancelled, reserve отпустилась полностью.
- Reverse POS-picking → компоненты возвращаются на склад (stock +1 на каждый).
- Chatter: «🗑 Разобран <флористом> через POS-NNN. Компоненты возвращены на склад.»

**Net stock:** 0. Reserve снят, физически ничего не двинулось.

#### 1.2.5. Архитектурный summary (v2)

| Сценарий | Method оплаты | Наш код | Штатный Odoo |
|---|---|---|---|
| 1. Собрать на витрину | «🌹 Собрать / изменить букет» (6) | Action 1203 branch create: SO BP-*, confirm → assigned, reverse POS-picking | — |
| 2. Модификация | «🌹 Собрать / изменить букет» (6) | Action 1203 branch reassemble: cancel old, create new BP-*, confirm → assigned, reverse POS-picking | — |
| 3. Продажа клиенту | Cash / Card / любой обычный | Action 1203 branch sell: только chatter | cancel SO-picking, POS-picking списывает, qty_delivered += 1 |
| 4. Разборка | «🗑 Разобрать букет» (8) | Action 1209: cancel old, reverse POS-picking | — |
| 5. Собрал+продал одним чеком | Cash / Card + маркер в корзине | Обычный POS-чек | Стандартный |

**Что не используется больше (v1 deprecated):**
- Product 7865 `[BQ-DISMANTLE]` — archived.
- Actions 1205 / 1207 — в базе, но их automations 11/12 disabled.
- Ветка dismantle в action 1203 — удалена, перенесена в action 1209.
- Cancel SO-picking после action_confirm — убран (ломал reserve-model).

#### 1.2.4. Две формы одного жизненного цикла

Букет может существовать в двух формах — это не «колебание решения», а **два валидных состояния одной учётной сущности** (инвариант 32):

| Форма | Когда используется |
|---|---|
| **Pre-built** — собран заранее, стоит на витрине | Готовые букеты «под расхватывание»: флорист делает утром, клиент покупает днём |
| **Made-on-sale** — собирается прямо при продаже клиенту | Персональные букеты «под заказ»: флорист собирает при клиенте |

Обе формы проходят через один lifecycle `Создан → [существует] → Разобран/Продан`. В MVP — реализовать хотя бы одну (приоритетно Made-on-sale как частотный POS-кейс), вторая дорастёт.

### 1.3. Приёмка товара

**Интерфейс:**
- Отдельный экран «Приёмка»
- **Push-уведомление** «к магазину едет товар» — чтобы флорист был готов
- Видит входящие поставки по его складу

**Идентификация позиций:**
- **По штрихкоду** (приоритет) — сканирование
- **Вручную** — если штрихкодов нет

**Фиксация количеств:**
- **Обязательный реальный пересчёт**, не слепое подтверждение накладной
- Раздельный ввод:
  - Пачки (`x_studio_received_packs`)
  - Штуки (`quantity`)
- Sentinel `-1` = «не пересчитано» ([см. 03](03_odoo_receipt_review.md))

**Фотофиксация:**
- Опция приложить **одно или несколько фото**:
  - Общий вид партии
  - Конкретная проблема (порча, недопоставка)
- Используется для контроля, разборов, качества поставок

**Расхождения:**
- Система подсвечивает (`review_status` + `review_color`)
- Автоматически по логике severity 0-3 ([03_*.md](03_odoo_receipt_review.md))

**Open question:** ⬜ Флорист видит только **свои** поставки или весь список — не решено.

### 1.4. Списание товара (с фото-согласованием)

**Никогда не мгновенная операция.** Всегда — процесс через утверждение.

**Процесс:**
1. **Флорист:**
   - Выбирает позиции на списание
   - Делает фото-доказательство
   - Указывает тип списания (см. ниже)
   - Отправляет на согласование
2. **Главный флорист / Manager (жена / директор):**
   - Получает notification
   - Проверяет
   - Подтверждает или отклоняет
3. После подтверждения → **списание со склада**

**Типы списания:**
- **Завял** (естественная порча)
- **Сломался** (механическое повреждение)
- **Потерян** (не найден в инвентаризации)
- **Ушёл в mix / пересортица** (списан с одной карточки, оприходован на другую)
- **Забыт к списанию** (списываем задним числом — индикатор процессной проблемы)

**Назначение процесса:**
- Контроль потерь
- Защита от ошибок и злоупотреблений
- Доказательная база для разборов

**Открытый вопрос:** ⬜ Может ли обычный флорист **сам** инициировать списание (с согласованием) или только главный флорист инициирует?

### 1.5. Задания от системы

**Принцип:** Odoo активно выдаёт магазину задачи, не только принимает действия.

**Типы заданий:**
- **Частичная инвентаризация:** «пересчитать розы / пересчитать именно эти 5 SKU»
- **Rolling correction:** «при первой продаже этой вазы — подтвердить остаток»
- **Срочная перепечатка ценника** (товар изменил цену → старый ценник на витрине устарел)
- **Фото-отчёт:** «помой холодильник + пришли фото», «проверь ценники наклеены — пришли фото»
- **Уведомление о прибытии товара:** «едет поставка №X, приготовься»

**Контроль выполнения:**
- Статус: pending / done / overdue
- Напоминание: «задача висит 3 часа, до сих пор не сделано»
- Планшет привлекает внимание к срочным задачам (всплывающий badge)

**Реализация:** через штатный Odoo `project.task` + кастомный UI → ⬜ в беклоге

### 1.7. Авторизация без полноценной лицензии

## 2. Рабочее место логиста

### 2.1. Закупка у поставщиков

**Сценарий:** логист едет на оптовую базу, покупает у нескольких поставщиков, фиксирует покупку.

**Требования:**
- Быстрый выбор поставщика
- Поиск товара — по SKU, названию, `x_studio_codigo_fabrica`, learned vendor code
- Указание количества в разных единицах (пачки / штуки)
- Опционально — сразу распределение по магазинам
- Привязка к своим кодам vs vendor codes

**Особенности:**
- Один товар у разных поставщиков — разные artikuly, разное количество в пачке, разная цена в разные дни
- **Подаренные единицы** — поставщик положил бонус → отдельный учёт
- Цена может меняться день ко дню

**Open questions:**
- ⬜ Прямо в момент закупки распределить по магазинам (часть в Plaza, часть в Gloria) — или отдельным документом?
- ⬜ Телефон vs планшет для логиста?

### 2.2. Связь с приёмкой флориста

**Принцип (формулировка пользователя):**
> *«Дорогой флорист, мы тут тебе помогли и уже посчитали… но ВСЁ РАВНО ВСЁ ПЕРЕСЧИТАЙ САМ»*

**Реализация:**
- `x_studio_expected_qty` на `purchase.order.line` — логист заполняет, если пересчитывал на месте/в пути
- Флорист видит это как **подсказку**, но обязан пересчитать сам
- В receipt видно 4 числа:
  - Что было на бумаге (`x_studio_paper_qty`)
  - В какой единице бумага (`x_studio_paper_unit`)
  - Что логист ожидал (`x_studio_expected_qty_info`)
  - Что флорист реально принял (`quantity`, `x_studio_received_packs`)

**Отвергнуто:**
- Boolean «логист сказал пачки» — лишнее, выражается через `uom_id = Paquete`

### 2.3. Контроль поставщиков

**Что фиксируется:**
- Недопоставки (paper > actual)
- Пересортица
- Расхождения по цене (paper vs factura / bill)
- Подаренные единицы
- Случаи, когда логист увидел проблему в пути ещё до магазина

**Три слоя разнесены:**
- Бумага поставщика (PO)
- Фактическая реальность (receipt)
- Логистическая оценка (expected_qty)

**Open questions:**
- ⬜ Отдельный отчёт «поставщики performance» или достаточно view по receipt / PO / invoice?
- ⬜ Фиксация **причины** расхождения (поставщик / транспорт / разбилось / простили) — не решено

### 2.4. Оплата / vendor bills

**Модель:**
- Албаран / PO / receipt — частые (еженедельные)
- Фактура от поставщика — **ежемесячная, закрывает все albaran за период**
- Для цветов/горшечки → `purchase_method='purchase'` (платим по бумаге, [03](03_odoo_receipt_review.md))

**Ответственность:**
- Логист видит состояние bill? ⬜ не решено
- Оплата — исключительно бухгалтер

### 2.5. Open questions логиста

- Глубина участия в multi-warehouse split
- Где заканчивается зона логиста — на вводе pedido / распределении / подтверждении по магазинам
- Финальный упрощённый UX на планшете

## 3. Рабочее место бухгалтера (три задачи)

### 3.1. Сверка продаж

**Что делает вручную сегодня:**
- Продажи в системе × отчёты магазинов в Telegram × касса × банковские поступления
- Сверяет построчно, ищет расхождения
- При проблеме — ищет причину в telegram-чате магазина

**Проблемы:**
- Продали сегодня → банк прислал деньги позже
- Часть транзакций пришла не сразу
- В системе одна цена → по терминалу сняли другую

**Куда автоматизировать:**
- `pos.session` ↔ `account.bank.statement` reconciliation (встроенное в Odoo)
- Возможно Make.com-бот, читающий telegram-отчёты и сверяющий с `pos.session.total`
- Аналитическая view: «несверенные смены» с прямыми ссылками

🔴 **CONCEPT** — не настроено.

### 3.2. Расходные накладные + испанская категоризация

**Что делает:**
- Вносит счета (аренда, услуги, коммуналка)
- Категоризирует по испанскому Chart of Accounts (PGCE PYMEs 2008)
- Оплачивает, собирает фактуры
- Результат → Modelo 303 / 347 / 349

**Куда автоматизировать:**
- `account.move` pipeline + AI-pre-categorization (на базе OCR-промпта из бота)
- `account.online.link` — bank connector
- Workflow: ожидание → получение → категоризация → оплата → прикрепление factura
- Quarterly AEAT reports check

🟡 **READY** — `l10n_es.*` локализация установлена, workflow нужно настроить.

### 3.3. Супервайзинг цикла

**Цикл в Holded требовал отдельного Google Sheets:**
```
закупили → привезли → приняли → пересчитали → распаковали →
выставили в продажу → напечатали ценники → проверили расхождения
```

**Контрольные точки:**
- Receipt (флорист принял)
- Склад (товар оприходован)
- Ценники (напечатаны, наклеены)
- Инвентаризация (периодическая)
- Задания магазину (выданы, выполнены)

**Куда автоматизировать:**
- **Kanban view** вместо Google Sheets → стадии pedido/товара
- Либо extended `stock.picking.state`
- Либо `x_studio_supervision_stage` selection на `purchase.order`
- Либо custom model с workflow

**Open:** ⬜ Кто именно контролирует шаг «распаковали → выставили»?

🔴 **CONCEPT** — не реализовано.

### 3.4. Что в Holded было неудобно

**Главные претензии:**
- Часть процессов — костыли
- Букет не моделируется как нужно
- Закупка / приёмка / фактическая реальность не разведены
- End-to-end супервайзинг требует внешнего Google Sheets
- Плоская архитектура не соответствует реальной операционке

**Вектор в Odoo:**
- **Бумага поставщика / ожидание логиста / факт флориста — отдельные слои**
- Сущности типа «букет» моделируются first-class, не через workarounds
- Полный жизненный цикл товара виден в одной системе

## 4. Модель оплаты и бонусы

**Статус:** 🟡 CONCEPT — модель согласована, **коэффициенты калибруются после сбора реальных данных**.

### 4.1. Структура зарплаты флориста

| Компонент | Сумма / правило |
|---|---|
| **Базовая ставка** | 900 € — гарантированно всегда |
| **Минимум гарантии** | 1400 € (официальная минимальная ЗП Испании) — если личные бонусы не дотягивают, доплачиваем до 1400, **но ставим флориста на карандаш** |
| **Личные бонусы** | От 0 до ∞ — всё сверх 1400 идёт сверху |

### 4.2. Что входит в базовые 900 € (не оплачивается дополнительно)

Вся **операционка магазина, не связанная с продажами:**
- Приём цветов, физическая приёмка
- Мытьё холодильников, чистка витрин, общая уборка
- Открытие / закрытие магазина
- Выполнение типовых системных заданий (пересчёт, фото-отчёты)
- Сборка букетов на витрину (без продажи)

### 4.3. Что даёт личный бонус

**Все продажи, привязанные к сотруднику:**
- Offline продажи через POS (в любом из 3 магазинов)
- Обработанные онлайн-заказы: Flowwow / Glovo / WhatsApp / Instagram / сайт
- Кто обрабатывал заказ → тот получает бонус с этого заказа

### 4.4. Ключевые принципы

- **Личные, не коллективные.** Флорист конкурирует, не делит смену.
- **Работает во всех каналах:** offline POS + online.
- **Индикатор проблем:** флорист стабильно получает только 900 € → флаг «не справляется» → разбор → возможно увольнение.

### 4.5. Калибровка

**Этап 1 (сейчас — запуск):** запускаем **без формул бонусов**, просто собираем данные:
- Сколько каждый продаёт (personal sales report)
- Сколько в среднем по точке
- Распределение по часам / дням / каналам

**Этап 2 (2-3 месяца работы):** анализируем реальные данные:
- Средняя производительность
- Верхний квартиль — какую премию должны получать
- Нижний квартиль — какой минимум мотивирует

**Этап 3:** моделирование коэффициентов бонусной формулы. Цель:
- Не переплатить (чтобы маржа магазина сохранилась)
- Не демотивировать (чтобы топ-флористы не чувствовали потолок)

**Этап 4:** запуск бонусной формулы во все каналы продаж.

### 4.6. Техническая предпосылка

Бонусная модель **невозможна без**:
- Привязки каждой продажи к `hr.employee` через смену
- PIN-авторизации в POS
- Personal sales report
- Единой воронки для online-заказов с идентификацией обработчика

Это всё → открытая работа ([09](09_open_work.md)).
